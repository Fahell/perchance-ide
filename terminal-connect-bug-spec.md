## 2. Problem Statement

When a user has registered a BrowserPod API key and opens the interactive terminal panel (via the `[term]` button in `EditorFooter.tsx`, the `Ctrl+\`` shortcut, or any other UI path), the panel displays:

```
‚ö† Failed to connect: Cannot read properties of undefined (reading 'catch')
```

A subsequent "Retry" button click does not recover ‚ÄĒ opening the panel a second time shows the same error directly without the intermediate "connecting‚Ä¶" spinner. Other BrowserPod-backed features (`run_npm_install`, `run_node_script`, `run_shell_command`, `start_http_server`) continue to work normally for the rest of the session.

The DevTools console does **not** show the offending `err.stack` anywhere visible to a non-technical user, but the user observed the earlier normal BrowserPod logs:

```
[Store] BrowserPod: booting ‚ÄĒ fresh initialization
[BrowserPod] Booting...
[BrowserPod] Ready
[BrowserPod] Custom .bashrc written to /home/user/.bashrc
```

`[BrowserPod] Custom .bashrc written to /home/user/.bashrc` confirming we reach *Step 2* in `createInteractiveTerminal()`. **The crash happens after `.bashrc` is written and before the bash spawn is handed to the user.**

## 4. Root Cause Analysis (thinker output + code review)

### Hypothesis A (highest confidence) ‚ÄĒ `/bin/true` health check consumes the interactive terminal

In commit `2e3101f` ("add interactive-terminal health check before starting bash"), the following block was inserted in `src/browserpod/manager.ts`, function `createInteractiveTerminal()` between the `.bashrc` write and the bash spawn:

```ts
// Step 3: Verify the interactive terminal is healthy BEFORE starting bash.
try {
  await this.pod.run("/bin/true", [], {
    terminal: this.interactiveTerminal,
    cwd: "/home/user",
  });
} catch (err) { ‚Ä¶ throw new Error("Interactive terminal WebSocket is broken: ‚Ä¶"); }
```

**Why this is the prime suspect**:
- `/bin/true` is a zero-input, zero-output program that exits in microseconds. When attached to a `Terminal` handle created seconds earlier by `createCustomTerminal({ cols, rows, onOutput })`, BrowserPod's internal bookkeeping for the terminal's stream/queued-output-promise can be torn down as soon as the process exits.
- The very next call (`pod.run("/bin/bash", ‚Ä¶, { terminal: this.interactiveTerminal, ‚Ä¶ })`) then attempts to reuse a terminal whose internal promise chain has already been finalized. BrowserPod internals do `somePromise.catch(...)` on a `terminatingPromise` that is now `undefined`, triggering the TypeError literally.
- The `err.message` propagated back to `TerminalPanel.tsx` is *exactly* `Cannot read properties of undefined (reading 'catch')` ‚ÄĒ a signature failing `.catch()` on undefined.

### Hypothesis B ‚ÄĒ Race between terminal creation and first `pod.run()`

`createCustomTerminal()` resolves with a handle, but BrowserPod's full duplex setup may not be done synchronously. If we run a command on the handle "before" the duplex is ready, internal promises can resolve to undefined.

- Mostly subsumed by hypothesis A: the *only* added timing pressure came from the `/bin/true` insertion.
- Standalone evidence: `cols`/`rows` are passed to `createCustomTerminal` but xterm.js sends resized values via `bridge.resize(cols, rows)` only after the bridge is wired up.

### Hypothesis C ‚ÄĒ Process handle promises `.then(...)` racing `.catch()`

The deeper `pod.run` returns `Promise<Process>`. In `manager.ts` Step 4 the manager attaches a fire-and-forget `.then(‚Ä¶).catch(‚Ä¶)`. If `pod.run` itself returns `undefined` synchronously (the documented guard `if (!runPromise || typeof runPromise.then !== "function") throw "pod.run() returned no promise";` is the worst-case branch), the *manager's own* code throws "Interactive shell could not be started ‚Ä¶". That's a different message, not the reported one ‚ÄĒ so C is **not** the cause of the user-visible error.

### Hypothesis D ‚ÄĒ On subsequent opens, `_bashrcWritten === true` fast-tracks the failure

The `await loadBashHistory()` in `TerminalPanel.tsx` + `.bashrc` write in manager both become near-instant on the second open. The user sees `connecting‚Ä¶` for a microtask flush, then the synchronous failure from `pod.run("/bin/true")`. Removing A removes the time pressure asymmetry and re-symmetrizes first/next open states.

### Root cause verdict

> **Hypothesis A is consistent with the symptoms, the recent commit history, and the structure of the failing await chain. Strategy: remove the `pod.run("/bin/true", ‚Ä¶)` "Step 3" interactive-terminal health check.** The original `_healthCheck()` in `manager.ts` (uses the *headless* terminal at lines ~110‚Äď135) already validates the BrowserPod WebSocket and is the correct layer for that check. Running a process through a freshly created `createCustomTerminal` handle is unnecessary and is the regression introduced in `2e3101f`.

## 5. Fix Strategy

### Primary recommendation ‚ÄĒ Option A (minimal & targeted)

Delete the entire **Step 3** `pod.run("/bin/true", ‚Ä¶)` block in `manager.ts::createInteractiveTerminal()`. Keep the headless `_healthCheck()` at the top of the function, which already catches the actually-interesting failure class (dead WebSocket / desynced pod).

```ts
// BEFORE (current ‚ÄĒ buggy):
// ‚ĒÄ‚ĒÄ Step 3: Verify the interactive terminal is healthy BEFORE starting bash.
try {
  await this.pod.run("/bin/true", [], {
    terminal: this.interactiveTerminal,
    cwd: "/home/user",
  });
} catch (err) {
  const msg = err instanceof Error ? err.message : String(err);
  console.warn("[BrowserPod] Interactive terminal health check failed:", msg);
  this.interactiveTerminal = null;
  throw new Error("Interactive terminal WebSocket is broken: " + msg + ‚Ä¶);
}

// Step 4: Start a bash shell ‚Ä¶
```

```ts
// AFTER (proposed):
// (Step 3 deleted ‚ÄĒ headless _healthCheck() above already proves the WebSocket is alive.)

// Step 3 (renumbered): Start a bash shell ‚Ä¶
```

**Comparison of candidate strategies:**

| Option | Lines changed | Risk to existing tests | Robustness vs. corner cases | Maintenance burden |
|---|---|---|---|---|
| **A** ‚ÄĒ drop the `/bin/true` block entirely | ~ -18 / ~ +2 | Low (lifecycle tests don't mock the interactive health branch) | Good for current browserpod 2.12.1; relies on headless `_healthCheck()` | Very low |
| **B** ‚ÄĒ swap `/bin/true` for `tty` / `sleep 0.01` | ~ -2 / ~ +0 | Medium (changes observable behavior in tests) | Higher risk of unintended side-effects (`tty` may print device-path to stdout that bleeds into xterm) | Medium |
| **C** ‚ÄĒ full retry-with-backoff + force-recreate | ~ +60 new code | High (test infrastructure needs fakes for retry timing) | Best long-term; overkill for this targeted regression | High |

User-preference signal: *"Quero que voc√™ pense mais antes de decidir"* ‚Üí presented A as the leading recommendation; user trusts deeper analysis to arrive at the right choice.

## 6. UX Additions (user-confirmed)

### 6.1 Collapsible stack trace on error UI

In `TerminalPanel.tsx`, when `error` is set, render the message plus a `<details>`-style block (or a manual toggle) showing the raw `err.name`, `err.message`, and `err.stack ? err.stack.split('\n').slice(0, 8).join('\n')` only when `__COMMIT__` endsWith `"dev"` (or another build flag we expose). This keeps production users safe from info-disclosure while letting developers and power users debug future regressions without re-clicking through DevTools.

### 6.2 "Reset BrowserPod" button

Add a `‚öô Reset BP` button next to the existing **Retry** and **Close** in the error-state row of `TerminalPanel.tsx`. Behavior:

1. Sets internal `resetting=true`, momentarily replaces buttons with a spinner and the message `'Rebooting BrowserPod‚Ä¶'`.
2. Calls `browserPodManager.dispose()` (best-effort, swallows errors).
3. Reads stored config from `ideStore.getState().settings.browserPodApiKey` (already exposed via the lifecycle PR `d4b850e`).
4. Calls `browserPodManager.boot({ apiKey, ‚Ä¶ })` with retry-key bump.
5. On success ‚Üí bumps `retryKey` so the existing `useEffect` re-runs init().
6. On failure ‚Üí restores error UI with a new error message.
7. **Edge cases**:
   - Double-click while rebooting: ignore second click via a local `inFlightRef`.
   - Other BrowserPod-using tools in flight: `dispose()` may interrupt them, but the same is true if the user toggles Node Tools off in Settings. Document the trade-off in the error message.

## 10. Implementation Outline (pseudocode)

### 10.1 `manager.ts::createInteractiveTerminal()` ‚ÄĒ simplified after fix

```ts
async createInteractiveTerminal(options) {
  if (!this.pod) throw new Error("BrowserPod not initialized");
  await this.disposeInteractiveTerminal();
  if (!this.pod) throw new Error("BrowserPod was disposed while initializing the interactive terminal");

  // Headless WS health check (kept ‚ÄĒ covers real failure class).
  const healthy = await this._healthCheck();
  if (!healthy) {
    throw new Error(
      "BrowserPod connection lost ‚ÄĒ the WebSocket may have timed out. " +
      "Close the terminal panel and reopen it, or disable/re-enable Node.js tools in Settings."
    );
  }

  this.interactiveTerminal = await this.pod.createCustomTerminal({
    cols: options.cols,
    rows: options.rows,
    onOutput: (buffer) => options.onOutput(new Uint8Array(buffer.slice(0))),
  });

  if (!this.pod) {
    this.interactiveTerminal = null;
    throw new Error("BrowserPod was disposed while creating the terminal");
  }
  if (!this.interactiveTerminal) {
    throw new Error("createCustomTerminal returned no terminal handle");
  }

  await this._ensureBashrc(); // extracted from current inline block

  // (DELETED ‚ÄĒ was the interactive-only /bin/true health check.)

  // Start bash directly.
  const runPromise = this.pod.run("/bin/bash", [], {
    terminal: this.interactiveTerminal,
    cwd: "/home/user",
    env: ["HOME=/home/user"],
  }) as Promise<unknown>;

  if (!runPromise || typeof (runPromise as any).then !== "function") {
    this.interactiveTerminal = null;
    throw new Error("Interactive shell could not be started ‚ÄĒ pod.run() returned no promise");
  }

  runPromise.then(() => console.log("[BrowserPod] Interactive shell exited"))
            .catch((err) => {
              const msg = err instanceof Error ? err.message : String(err);
              console.warn("[BrowserPod] Interactive shell failed:", msg);
              this.interactiveTerminal = null;
            });

  return makeBridge(this); // unchanged
}
```

### 10.2 `TerminalPanel.tsx` ‚ÄĒ Reset BP state machine

```ts
const [resetting, setResetting] = useState(false);

async function handleResetBrowserPod() {
  if (resetting) return;
  setResetting(true);
  setError(null);
  try {
    await browserPodManager.dispose().catch(() => {});
    const apiKey = ideStore.getState().settings.browserPodApiKey;
    if (!apiKey) throw new Error("API key not configured");
    const ok = await browserPodManager.boot({
      apiKey,
      storageKey: "agent-perchance",
      nodeVersion: "22",
    });
    if (!ok) throw new Error(browserPodManager.getError() ?? "boot failed");
    await browserPodManager.subscribeToVfsChanges();
    setRetryKey((k) => k + 1); // re-run init()
  } catch (e) {
    setError(`Failed to reset BrowserPod: ${e instanceof Error ? e.message : String(e)}`);
  } finally {
    setResetting(false);
  }
}

// Error UI row gets a third button:
{!resetting && (
  <button onClick={handleResetBrowserPod}>‚öô Reset BP</button>
)} {resetting && <span>Rebooting BrowserPod‚Ä¶</span>}
```

## 12. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Fix is wrong (root cause is actually B or C) | The fix is contained to non-critical-path code (a process was being run purely as a check). Worst case: loss of one pre-flight signal that was redundant with `_healthCheck()`. |
| Bash starts but WebSocket was actually dead | `_healthCheck()` already exercises the same kind of round-trip via the headless terminal. If we want extra safety, can move to Option B later. |
| `Reset BP` disposes an in-flight `npm install` | (a) Documented in the error UI copy. (b) We can add a `confirm()` prompt blocking browser-tool calls before dispose. |
| Dev build flag (`__COMMIT__` ending `"dev"`) mis-classifies some production commits | Use `__VERSION__` containment or a more explicit `process.env.NODE_ENV` injection if we ever worry. Currently fine because CI never publishes a `@dev` commit to `dist`. |
